# Naver Blog Writer — 5팀 파이프라인

## Overview

이 skill은 교육법 전문 변호사(염용주 대표변호사) 페르소나로 네이버 블로그에 최적화된 법률 정보 글을 작성한다. 5개 전문팀 에이전트가 순차적으로 작업하여, 기획→리서치→작성→SEO 최적화→검수를 거친 고품질 블로그 글을 생산한다. 최종 산출물은 마크다운 파일로 저장하고 Google Docs에도 업로드한다.

**산출물 기준:** 본문 2,500~3,000자(공백 제외), 제목 5개, 이미지 프롬프트 5개, 메타태그 15개+, 변호사 광고에 관한 규정 준수

## Pipeline

### Step 0: 글 유형 선택

사용자에게 다음 3가지 글 유형을 제시하고 선택받는다. 선택 결과를 `{content_type}` 변수에 저장하여 후속 Step들에 전달한다.

| 모드 | 타깃 단계 | 핵심 구조 | 적합한 주제 예시 |
|------|----------|----------|----------------|
| **전문 정보** (기본) | 탐색 | 개념→법적근거→절차→대응 | "교원 징계 절차와 소청심사 대응법" |
| **초입문자 가이드** | 인식 | 하지말것→해야할것→왜→다음단계 | "아이가 학폭 통보를 받았는데 어떡하죠?" |
| **성공 사례** | 결정 | 상황→쟁점→전략→결과→시사점 | "교원 징계 취소 사례로 보는 소청심사 핵심" |

**생략 조건:** 사용자가 주제와 함께 유형을 명시적으로 지정한 경우(예: "성공 사례로 써줘") 이 단계를 생략하고 해당 유형으로 진행한다.

→ 선택 결과를 `content_type` 변수에 저장 (전문 정보 / 초입문자 가이드 / 성공 사례)

### Step 1: 기획팀 (blog-planner)

사용자 주제를 분석하여 콘텐츠 플랜을 수립한다.

```
Task(blog-planner):
  prompt: |
    주제: {사용자 주제}
    글 유형: {content_type}
    오늘 날짜: {오늘 날짜}

    위 주제에 대해 콘텐츠 플랜을 수립하세요.
    글 유형에 맞는 아웃라인과 제목 패턴을 적용하세요.
    레퍼런스 파일을 반드시 참조하세요.
```

**입력:** 사용자 주제, 오늘 날짜
**출력:** 키워드 전략, H2/H3 아웃라인, 이미지 배치 계획, 제목 방향성, 리서치 요청사항

→ 출력을 `plan` 변수에 저장

#### [선택] 사용자 확인 포인트

기획안이 완성되면, 사용자에게 콘텐츠 플랜을 보여주고 확인을 요청한다:
- "콘텐츠 플랜이 완성되었습니다. 검토 후 진행하시겠습니까?"
- 사용자가 수정을 요청하면 기획팀을 재호출하여 반영
- 사용자가 승인하면 Step 2로 진행
- **빠른 실행 모드**: 사용자가 주제와 함께 "바로 작성해줘" 등 즉시 진행을 요청한 경우, 이 확인 단계를 생략하고 자동으로 Step 2로 진행한다

### Step 2: 리서치팀 (blog-researcher)

기획팀의 플랜을 바탕으로 최신 법률/판례/통계/뉴스를 조사한다.

```
Task(blog-researcher):
  prompt: |
    다음 콘텐츠 플랜에 대해 리서치를 수행하세요.
    글 유형: {content_type}
    레퍼런스 파일을 반드시 참조하세요.

    ## 콘텐츠 플랜
    {plan}
```

**입력:** 기획팀 콘텐츠 플랜
**출력:** 최신 법률, 판례 요약, 통계 데이터, 뉴스, 제작팀 활용 포인트

→ 출력을 `research` 변수에 저장

### Step 3: 제작팀 (blog-writer)

기획과 리서치를 바탕으로 블로그 초안을 작성한다.

```
Task(blog-writer):
  prompt: |
    다음 콘텐츠 플랜과 리서치 노트를 바탕으로 블로그 초안을 작성하세요.
    글 유형: {content_type}
    글 유형에 맞는 작성법을 적용하세요.
    레퍼런스 파일을 반드시 참조하세요.

    ## 콘텐츠 플랜
    {plan}

    ## 리서치 노트
    {research}
```

**입력:** 기획팀 플랜 + 리서치팀 노트
**출력:** 완성 블로그 초안 (2,500~3,000자), 제목 5개, 이미지 프롬프트 5개

→ 출력을 `draft` 변수에 저장

### Step 4: SEO팀 (blog-seo-optimizer)

블로그 초안에 네이버 알고리즘 최적화를 적용한다.

```
Task(blog-seo-optimizer):
  prompt: |
    다음 블로그 초안을 SEO/AEO 최적화하세요.
    글 유형: {content_type}
    레퍼런스 파일을 반드시 참조하세요.

    ## 핵심 키워드
    {plan에서 추출한 키워드}

    ## 블로그 초안
    {draft}
```

**입력:** 제작팀 초안 + 핵심 키워드
**출력:** 키워드 밀도 보고서, 최적화 본문, 메타태그 15개+

→ 출력을 `optimized` 변수에 저장

### Step 5: 검수팀 (blog-qa-reviewer)

최종 품질 검수를 수행하고 승인된 본문을 출력한다.

```
Task(blog-qa-reviewer):
  prompt: |
    다음 SEO 최적화 본문을 검수하세요.
    글 유형: {content_type}
    글 유형에 맞는 QA 항목을 적용하세요.
    레퍼런스 파일을 반드시 참조하세요.

    ## SEO 최적화 본문 + 메타태그
    {optimized}
```

**입력:** SEO 최적화 본문 + 메타태그
**출력:** QA 보고서 (글자수/광고규정/오탈자), 최종 승인 본문, 품질 검토 결과

→ 출력에서 **종합 판정**, **최종 승인 본문**, **품질 검토 결과**를 추출

### Step 5-1: QA 실패 시 재작업

검수팀의 종합 판정이 **"⚠️ 수정 필요"**인 경우 다음 절차를 따른다:

1. **경미한 수정** (오탈자, 표현 수정, 분량 미세 조정) — 검수팀이 직접 수정하여 최종 본문에 반영 후 승인 처리. Step 6으로 진행
2. **중대한 수정** (구조 변경, 핵심 내용 누락, 법률 사실 오류) — 다음 재작업 루프를 실행:
   - QA 보고서의 "수정 필요 사항"을 포함하여 **제작팀(blog-writer)**을 재호출
   - 제작팀 재작성 → SEO팀 재최적화 → 검수팀 재검수 (Step 3→4→5 반복)
   - 재작업은 **최대 1회**까지 허용. 2차 검수에서도 중대한 수정이 필요하면, 검수팀이 직접 수정하여 최종 승인 처리

```
[QA 판정이 "수정 필요"인 경우]

경미한 수정 → 검수팀 직접 수정 → Step 6 진행
중대한 수정 → Step 3(제작팀) → Step 4(SEO팀) → Step 5(검수팀) 재실행 (최대 1회)
```

### Step 6: 파일 저장 + Google Docs 업로드

검수팀이 승인한 최종 본문을 저장한다.

#### 6-1. 마크다운 파일 저장

**파일명 생성 규칙:**
1. 주제에서 핵심 키워드 2~3개를 추출한다 (예: "교원 성비위 징계" → `교원성비위징계`)
2. 키워드를 언더스코어(`_`)로 연결한다
3. 접미사 `_blog_YYYYMMDD.md`를 붙인다 (YYYYMMDD는 오늘 날짜)
4. 공백과 특수문자(쉼표, 물음표 등)는 제거한다

**파일명 예시:**
- "교원 성비위 징계, 무혐의인데도 징계받을 수 있나요?" → `교원성비위징계_무혐의징계_blog_20260219.md`
- "학교폭력 가해 학생 전학 조치, 불복 가능할까요?" → `학교폭력_가해학생전학_불복_blog_20260219.md`
- "2026 교육법 개정 핵심 5가지" → `2026교육법개정_핵심5가지_blog_20260219.md`

**저장 경로:** 프로젝트 루트 디렉토리 (`C:\workspace\blog-post\`)

최종 승인 본문 전체를 마크다운 파일로 저장한다. 본문, 추천 제목, 이미지 프롬프트, 메타태그, 품질 검토 결과를 모두 포함한다.

#### 6-2. Google Docs 업로드

1. **도구 로드**: `ToolSearch({ query: "+google-docs create document" })`
2. **마크다운 → plain text 변환**: 마크다운 파일 내용에서 아래 규칙에 따라 문법을 제거한다
   | 마크다운 문법 | 변환 | 예시 |
   |-------------|------|------|
   | `# `, `## `, `### ` | 제거 (텍스트만 유지) | `## 소제목` → `소제목` |
   | `**텍스트**` | `**` 제거 (텍스트만 유지) | `**중요**` → `중요` |
   | `*텍스트*` | `*` 제거 | `*강조*` → `강조` |
   | `---` (수평선) | 빈 줄로 대체 | |
   | `<!-- 주석 -->` | 그대로 유지 (이미지 배치 참고용) | |
   | `- ` (리스트) | 그대로 유지 | |
   | `1. ` (번호 리스트) | 그대로 유지 | |
   | 테이블 (`\| ... \|`) | 그대로 유지 | |
3. **문서 생성**: `mcp__google-docs__createDocument`로 새 문서 생성
   - title: 블로그 글 제목
   - initialContent: 마크다운 문법을 제거한 plain text 버전
4. **완료 확인**: 생성된 Google Docs 문서 URL을 사용자에게 제공

**Google Docs 업로드 실패 시 fallback:**
- ToolSearch에서 google-docs 도구를 찾을 수 없는 경우 → 마크다운 파일 저장만 수행하고, "Google Docs 연동이 설정되지 않았습니다. 마크다운 파일만 저장되었습니다."라고 안내
- createDocument 호출이 실패한 경우 → 1회 재시도 후에도 실패하면, "Google Docs 업로드에 실패했습니다. 마크다운 파일은 정상 저장되었습니다."라고 안내하고 파이프라인 완료 처리
- Google Docs 업로드 실패가 전체 파이프라인의 실패를 의미하지는 않는다

**Google Docs 포함 내용:**
- 추천 제목 (5개)
- 블로그 본문 (서론, 본론 3개, 결론)
- 이미지 생성 프롬프트 (영문) 5개
- 메타태그 (해시태그) 15개 이상
- 품질 검토 결과
